# Matrix (Conduit) redeploy kit — Ubuntu VPS + Caddy

This deploy kit is intentionally **non-secret**. Secrets are referenced as placeholders and should live in Bitwarden Secrets Manager.

Target hostnames:
- Homeserver: `matrix.gruted.us.com`

---

## 0) Prereqs (root)

```bash
apt-get update
apt-get -y install ca-certificates curl git ufw jq

# Docker
install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
chmod a+r /etc/apt/keyrings/docker.gpg
. /etc/os-release
printf "deb [arch=%s signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu %s stable\n" "$(dpkg --print-architecture)" "$VERSION_CODENAME" \
  > /etc/apt/sources.list.d/docker.list
apt-get update
apt-get -y install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Caddy
apt-get -y install debian-keyring debian-archive-keyring apt-transport-https
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | tee /etc/apt/sources.list.d/caddy-stable.list
apt-get update
apt-get -y install caddy
```

Firewall (adjust if you already have UFW rules):

```bash
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow 8448/tcp
ufw enable
ufw status
```

## 3) Render secrets into a deploy-time Conduit config

This repo contains `conduit.toml` with `${PLACEHOLDER}` variables.

### Option A (recommended here): use `matrix-secrets-env`

If you have the helper script (see `notes/matrix-bringup.md`), it will:
- pull required values from Bitwarden Secrets Manager
- write runtime-only files under `/run/user/<uid>/matrix` with `0600`

```bash
# Do NOT print secret values to stdout.
# MATRIX_BWS_PROJECT_ID is your Bitwarden Secrets Manager project id.
MATRIX_BWS_PROJECT_ID="<your-bws-project-id>" \
  ~/.local/bin/matrix-secrets-env

# The rendered TOML (contains secrets):
#   /run/user/<uid>/matrix/conduit.rendered.toml
```

### Option B: envsubst (manual)

See the older `envsubst` approach in previous revisions if you prefer.

## 4) Start Conduit

`docker-compose.yml` expects an environment variable `CONDUIT_RENDERED_TOML` pointing at your **rendered** TOML.

If you start docker-compose via `sudo`, make sure the env var is passed through:

```bash
cd /opt/matrix

# Example path when generated by matrix-secrets-env as user gru (uid 1001)
sudo CONDUIT_RENDERED_TOML=/run/user/1001/matrix/conduit.rendered.toml \
  docker compose pull

sudo CONDUIT_RENDERED_TOML=/run/user/1001/matrix/conduit.rendered.toml \
  docker compose up -d

sudo docker logs -f conduit
```

Sanity check (loopback):

```bash
curl -sS http://127.0.0.1:8008/_matrix/client/versions | jq
```

## 5) Configure Caddy

Use the provided `deploy/matrix/Caddyfile` as a reference and add the `matrix.gruted.us.com` + `:8448` site blocks to your host Caddyfile.

```bash
# Backup existing
cp -a /etc/caddy/Caddyfile /etc/caddy/Caddyfile.bak.$(date -u +%Y%m%dT%H%M%SZ)

# Edit and add the matrix.gruted.us.com blocks
nano /etc/caddy/Caddyfile

caddy fmt --overwrite /etc/caddy/Caddyfile
systemctl reload caddy
systemctl status caddy --no-pager
```

Sanity checks:

```bash
curl -sS https://matrix.gruted.us.com/ | head
curl -sS https://matrix.gruted.us.com/_matrix/client/versions | jq

# Federation port
curl -sS https://matrix.gruted.us.com:8448/_matrix/federation/v1/version | jq
```

## 6) Create an admin user

Conduit supports admin creation / registration flows that can vary by version.

Practical approach:
1) Temporarily enable **token-gated registration** in the rendered TOML:
   - `allow_registration = true`
   - `registration_token = "..."` # Prefer key: MATRIX_CONDUIT_REGISTRATION_TOKEN from Bitwarden SM instead of legacy matrix/prod path
2) Restart Conduit:

```bash
sudo docker compose restart conduit
```

3) Register the admin user using a Matrix client (Element) using the token.
4) Disable registration again and restart Conduit.

## 7) Backups (signing keys!)

**Back up the entire Conduit data volume**. It contains the homeserver signing keys (critical for federation identity), media, and state.

Example backup (stop briefly for consistency):

```bash
cd /opt/matrix

sudo docker compose stop conduit

sudo docker run --rm \
  -v conduit_data:/data:ro \
  -v /root:/backup \
  alpine sh -c 'cd /data && tar -czf /backup/conduit_data_$(date -u +%Y%m%dT%H%M%SZ).tar.gz .'

sudo docker compose start conduit

ls -lh /root/conduit_data_*.tar.gz
```

Store the resulting tarball securely (e.g., Bitwarden secure file / offline backup).

## Notes / options

### Do we need `.well-known`?

If your server name is **exactly** `matrix.gruted.us.com`, clients will use that domain in MXIDs (e.g. `@you:matrix.gruted.us.com`). In that case `.well-known` on `gruted.us.com` is optional.

If you want **`@you:gruted.us.com`** while hosting on `matrix.gruted.us.com`, then you should add `.well-known/matrix/client` and `.well-known/matrix/server` responses on `gruted.us.com` (see the optional Caddyfile snippet).
